# Auditoria tecnica y funcional - Palestra APP

Fecha: 2026-05-24

## Riesgos antes de una Beta real

- `App.tsx` concentra demasiada logica de pantallas, panel dirigencial, formularios y render. Riesgo: cambios chicos rompen modulos no relacionados. Conviene separar por modulo antes de crecer mucho mas.
- Hay fallbacks locales en `src/data/content.ts`. Son utiles offline, pero hoy pueden confundirse con contenido real si Supabase no devuelve datos.
- Varias funciones RPC viven como parches SQL acumulados en `supabase/*.sql`. Riesgo: una base nueva queda incompleta si no se ejecutan en orden.
- Deep link depende de configuracion externa de Supabase Dashboard. Si `Site URL` o allowlist quedan en localhost, la app no puede corregirlo desde codigo.
- Las pantallas admin dependen de permisos UI y RPC/RLS. Hay que seguir auditando que cada accion sensible tenga RPC segura, no solo boton oculto.

## Login y registro

- Archivo principal: `App.tsx`.
- El flujo nuevo usa `SafeAreaView`, pero la X estaba absoluta con `top: 16`. Se corrigio con `insets.top + 12`.
- Registro usa `emailRedirectTo` hacia `palestra://auth/callback`. El preview `?preview=mail-confirmed` queda solo para prueba visual.
- Recuperacion usa `redirectTo` hacia `palestra://auth/callback?flow=password-reset`.
- Falta una pantalla completa de cambio de contrasena post recovery. Por ahora no se rompe el login, pero el flujo ideal seria una vista especifica.

## Roles y permisos

- Logica central: `src/lib/roles.ts`, `src/lib/permissions.ts`, `src/lib/authProfile.ts`.
- Permisos dinamicos pasan por `getDynamicPermissionsForRole`, con fallback local.
- Riesgo: si `role_permissions` no tiene datos, el fallback local puede ocultar problemas reales de permisos.
- Roles alias y etiquetas existen, pero conviene unificar en una pantalla admin unica para no duplicar criterios.

## Comunidades

- Lectura publica: `src/lib/remoteData.ts` con `fetchCommunities()`.
- Edicion admin: `src/lib/profiles.ts` con `createCommunity`, `updateCommunity`, `archiveCommunity`.
- Imagenes: se usan `image_url` y Storage de Supabase.
- Riesgo: mapas/coordenas quedaron como estructura posible, pero no conviene reactivar editor de mapa hasta que comunidades quede estable varias tandas.

## Noticias

- Noticias internas: `src/lib/remoteData.ts` con `fetchNews()`, tabla `news`.
- Creacion: `src/lib/profiles.ts` con `createNews()`.
- Noticias externas: `src/lib/externalNews.ts`, fuente RSS Vatican News. No se mezcla con noticias internas.
- AICA queda como fuente evaluada, pero no integrada porque no se confirmo RSS estable desde codigo. Se puede sumar luego si publican endpoint RSS/API claro.

## PM

- Admin: `App.tsx`, modulo `periodo_motivador`.
- Persistencia: `src/lib/profiles.ts` con RPC `admin_upsert_motivador_period`.
- Lectura publica: `src/lib/remoteData.ts` con `fetchMotivadorPeriods()`.
- Correccion actual: filtros compactos por provincia, tipo, estado, fecha y año.
- Riesgo: PM publico sigue usando `flatMap` por fecha en `fetchMotivadorPeriods`; el listado visible agrupa con `groupMotivadorFeedItems`, pero cualquier nueva pantalla debe respetar ese agrupador.

## Descargas

- Lectura: `src/lib/profiles.ts` con `fetchAppMaterials()`, tabla `materials`.
- Guardado: `saveAppMaterial()` via RPC.
- UX pendiente: mostrar mejor estados de permiso y descarga fallida.

## Navegacion

- Lectura: `fetchAppTabs()` sobre `app_tabs`.
- Contenido: `fetchAppContent()` sobre `app_content`.
- Tipos reales agregados: simple, library, links, image_text, form, internal.
- Riesgo: `internal` depende de que el bloque `modulo` apunte a una key existente.

## Perfil y solicitudes

- Perfil real: `getMyProfileSession()` via RPC `get_my_profile`.
- Solicitudes: `fetchMyRequests()` y `fetchAdminRequests()`.
- Pendiente: mejorar estados vacios y separar mejor solicitudes del resto del perfil.

## Notificaciones

- Codigo principal: `requestAndRegisterPushToken`, `createNotificationIntent`, `deliverNotificationIntent`.
- Funciona con token Expo y canal Android.
- Riesgo: no tocar icono/canal salvo pruebas en dispositivo fisico, porque es sensible a assets nativos.

## Tema dark y responsive

- Tema: `src/theme/themes.ts` y multiples estilos condicionales en `App.tsx`.
- Inconsistencia: hay textos con estilos claros en cards blancas dentro de dark mode. Conviene migrar gradualmente a componentes base.
- Responsive: la app funciona mejor en mobile vertical; paneles admin largos necesitan mas dropdowns y menos chips.

## Pull to refresh

- Control: `RefreshControl` nativo en `App.tsx`.
- Refresca tabs, contenido, config y sesion.
- Algunas pantallas hacen fetch interno con `refreshKey`; si una pantalla no depende de `refreshKey`, puede quedar stale.

## Que agregaria

- Tabla `app_runtime_config` con minVersion, recommendedVersion, maintenanceMode y feature flags.
- Pantalla admin de salud del sistema: SQL aplicado, version de app, estado de Storage, permisos RPC.
- Separacion de `App.tsx` por modulos antes de sumar mas herramientas.
- Pruebas e2e manuales documentadas para: login, registro, comunidades, PM, descargas, push y refresh.
